# project-structure-standard — Python 项目全链路规范

## 一句话

散乱项目按统一模板重组 + 每个文件写清接口契约和调用链，AI 只读一个文件就能安全改代码。

---

## 适用场景

- 项目根目录几十上百个文件混在一起，分不清生产代码和调试产物
- AI 改了一个文件，不知道会不会改坏其他模块
- 你要快速找到：入口在哪、配置在哪、核心逻辑在哪
- 多个项目需要统一的目录和代码规范

## 解决的问题

| 问题 | 方案 |
|------|------|
| 文件散落根目录 | 按职责划分子目录 |
| 改一个文件改坏其他模块 | 接口契约锁定签名+数据格式 |
| 不知道谁调了谁 | 调用链追溯标注 |
| 配置散落各处 | 集中到 `config.py` |
| 调试探针混入生产 | 隔离到 `dev/` |
| 不知道哪里能改 | `★` 标记标明修改点 |

## 一、通用目录模板

```
项目名/
│
├── core/                  ★ 不动：核心框架/基类
├── adapters/              ★ 不动：平台适配器
├── post/                  ★ 不动：后处理/回调
├── dashboard/             ★ 不动：看板/可视化
│
├── tools/                 ★ 独立功能工具
│
├── bin/                   ← 批处理/快捷脚本
├── config/                ← 配置文件（密钥、cookies）
│
├── dev/                   ← 调试探针/截图（不参与生产）
│   ├── probes/            ← _probe*.py
│   ├── debug/             ← debug*.py
│   ├── records/           ← 运行记录
│   ├── explores/          ← explore*.py test*.py
│   └── screenshots/       ← *.png
│
├── main.py                ★ 调度入口
├── config.py              ★ ★ ★ 你改配置的地方 ★ ★ ★
│
├── data/                  ← 数据文件（gitignore）
├── logs/                  ← 运行日志（gitignore）
│
└── README.md
```

### 核心规则

1. **根目录只放调度入口** — `main.py` 是唯一根级脚本
2. **配置只改 `config.py`** — 路径、参数、开关全部集中
3. **调试文件不进根目录** — 一律放入 `dev/`
4. **★ 标记告诉你哪里能改** — 带 `★` 的文件是你会碰到的
5. **后处理归入 `post/`**，独立功能归入 `tools/`

---

## 二、★ 标记系统

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `★ ★ ★` | 唯一配置修改点 | `config.py` 里的配置变量 |
| `★` | 看懂了可以改 | `main.py` 的调度函数 |
| `★ 不动` | 框架代码别碰 | `core/` `adapters/` 等 |
| `╳ 不要动` | 绝对不要改 | 接口签名、数据结构 |

## 三、文件头部规范

### 3.1 头部大框（每个 .py 文件必须有）

```
╔══════════════════════════════════════════╗
║ 文件名.py — 一句话说明                    ║
║                                           ║
║ 调用关系：被谁调用 → 本文件 → 调用了谁    ║
║ 数据流向：输入从哪来 → 处理 → 输出到哪去  ║
║                                           ║
║ ★ 接口契约：                              ║
║   函数名(入参) → 出参                     ║
║   入参格式 / 出参格式                     ║
║                                           ║
║ ★ 调用链：                                ║
║   导出函数 → 被谁调用                     ║
║   引用了哪些外部函数                      ║
║                                           ║
║ ★ 能改的区域：xxx                         ║
║ ╳ 不能动的区域：xxx                       ║
╚══════════════════════════════════════════╝
```

### 3.2 变更记录（大框之后、代码之前）

```python
# ═══════════════════════════════════════
# 变更记录
# ═══════════════════════════════════════
# 日期       | 作者 | 改了啥
# ───────────────────────────────────────
# 2026-06-05 | AI   | 初始版本
```

每次修改追加一行，禁止覆盖历史。

### 3.3 全文逻辑说明（大框之后、变更记录之前）

一段中文，说清这个文件的整体逻辑，包括：这个文件做了什么、调用上下游是谁、数据怎么流经这个文件。

```python
# 这个文件负责从抖音来客导出成交明细。
# 被 main.py 的 跑() 调用，读取 config.py 的配置。
# 流程：登录 → 选日期 → 选导出类型 → 提交 → 等待 → 下载。
# 输入：config.py 的 cookies 路径、日期范围
# 输出：Excel 文件保存到 config.py 指定的目录
```

### 3.4 分步标记

```python
# ═══════════════════════════════════════════════════
# 第X步：这段代码在干嘛
# ═══════════════════════════════════════════════════
```

## 四、代码风格

### 4.1 import 写注释 + 三组分隔

注释说这个库在**当前文件**里具体干嘛用。三组之间空一行。

```python
# 标准库
import os          # 操作用户文件夹
import sys         # 控制搜索路径

# 第三方库
import pandas as pd    # DataFrame 处理导出数据
from playwright.async_api import async_playwright  # 浏览器自动化

# 本项目模块
from core import BaseExtractor   # 适配器基类
from config import 导出目录       # 全局配置
```

### 4.2 函数写中文 docstring

```python
def 函数名(参数1, 参数2):
    """
    一句话说明干嘛。

    参数：
        参数1：含义，示例值
        参数2：含义，示例值

    返回：返回什么，什么格式

    举例：函数名(值1, 值2) → 返回什么
    """
```

### 4.3 变量命名：中英混用

| 层次 | 用中文 | 用英文 |
|------|--------|--------|
| 业务逻辑 | ✅ 函数名、变量名、参数名 | ❌ |
| Python 语法 | ❌ | ✅ 关键字、内置函数、特殊方法 |
| 第三方库调用 | ❌ | ✅ 方法名、属性名 |
| 数据/配置 | ❌ | ✅ SQL、键名、列名 |

```python
def 取数据(配置, 日期):
    sql = "SELECT * FROM 销售表"
    原始数据 = pd.read_sql(sql, 配置["连接"])  # 库方法保留英文
    return 原始数据.groupby("店铺").sum()
```

### 4.4 错误处理

```python
# ✅ 正确：捕获具体异常 + 中文信息 + 记日志
try:
    await 下载文件(page, url)
except TimeoutError:
    logger.error(f"下载超时: {url}")
    return None  # 返回安全的默认值，不要 raise 中断整个流水线
except Exception as e:
    logger.error(f"下载失败: {url}, 原因: {e}")
    return None

# ❌ 错误：裸 except 吞掉一切
try:
    ...
except:
    pass

# ❌ 错误：没有中文信息
except Exception as e:
    print(e)
```

**原则：**
- 流水线类（导出/日报）：捕获异常 → 日志 → `return None`，不中断后续步骤
- 工具类/一次性脚本：可以 raise，但要写清 raise 的条件
- 禁止裸 `except:` ，至少写 `except Exception`
- 每个 except 块必须有中文说明

### 4.5 类型标注

```python
# ✅ 公共接口必写：基类方法、适配器入口、工具函数
async def 提取数据(self, 日期: str, 门店: str) -> list[dict]:
    ...

# ✅ 内部辅助函数可以省略
def _拼URL(基础, 参数):
    ...

# ❌ 全部不写 —— AI 读代码时类型标注是关键上下文
async def 提取数据(self, 日期, 门店):
    ...
```

**原则：** 被其他模块调用的函数（★标记的）必须写，文件内部的私有函数（`_` 开头）可以省略。

### 4.6 函数体约束

| 规则 | 限制 | 超了怎么办 |
|------|------|-----------|
| 单个函数不超过 50 行 | 超过就拆子函数 | `_步骤1_xxx()` `_步骤2_xxx()` |
| 嵌套不超过 3 层 | if/for/try 再深就难读 | 用早返回（见 4.9） |
| 单个文件不超过 300 行 | 超了说明职责不够单一 | 拆模块 |

### 4.7 魔法值 → 常量

```python
# ❌ 数字/字符串散落在代码里
await page.wait_for_timeout(30000)
if 门店数 > 160:
    ...

# ✅ 提到文件顶部或 config.py
_超时等待 = 30000      # 30秒
_门店上限 = 160        # 账号数上限
```

**原则：** 出现两次以上的数字/字符串，或者有业务含义的值（阈值、上限、超时），必须提取为常量。

### 4.8 日志规范

```python
import logging
logger = logging.getLogger(__name__)

# ✅ 用 logger，不用 print
logger.info(f"开始导出: {日期} {门店}")
logger.warning(f"门店数超限: {门店数} > {_门店上限}")
logger.error(f"下载失败: {url}")

# 级别用法：
# DEBUG — 调试探针信息（默认不输出）
# INFO  — 正常流程节点（"开始导出"、"导出完成"）
# WARNING — 可恢复的异常（"重试第2次"）
# ERROR — 导致当前步骤失败（"下载超时"）
```

**原则：** 生产代码一律 `logger`，`print` 只在临时探针脚本里用。

### 4.9 早返回（卫语句）

```python
# ❌ 层层嵌套
async def 导出(日期, 门店):
    if 日期:
        if 门店:
            数据 = await 取数据(日期)
            if 数据:
                return 保存(数据)

# ✅ 不满足条件就早返回
async def 导出(日期, 门店):
    if not 日期 or not 门店:
        logger.warning("缺少日期或门店参数")
        return None
    数据 = await 取数据(日期)
    if not 数据:
        logger.warning(f"无数据: {日期}")
        return None
    return 保存(数据)
```

### 4.10 禁止事项

- ❌ 函数没有 docstring
- ❌ 配置硬编码在代码里
- ❌ 调试文件放在根目录
- ❌ try 块里没有中文错误信息
- ❌ 一段逻辑从头到尾没有注释
- ❌ 裸 `except:` 吞掉所有异常
- ❌ `print()` 写在生产代码里
- ❌ 魔法数字（有业务含义的数字必须提取为常量）
- ❌ 超过 50 行的函数不拆分
- ❌ 嵌套超过 3 层

## 五、接口契约（零上下文修改的兼容性保障）

### 核心原则

每个文件的大框里标注接口契约，AI 只读一个文件就知道哪些能改、改了会不会破坏其他模块。

### 三层保障

| 层 | 保障什么 | 怎么写 |
|----|----------|--------|
| 第1层 | 函数签名不变 | 大框标注 `def 函数名(参数)` → 只能改函数体 |
| 第2层 | 数据格式不变 | 大框标注列结构：`行[0]=日期, 行[1]=门店` |
| 第3层 | 行为兼容验证 | `dev/probes/_probe_check_compatibility.py` |

### 改动风险等级

| 改动类型 | 风险 | 预防 |
|----------|------|------|
| 改函数体内部逻辑 | 低 ✅ | 接口契约锁定签名 |
| 改函数名/入参/返回格式 | 高 ❌ | 大框标注"不能改" |
| 改配置键名 | 高 ❌ | config.py 头部写死"不能改" |
| 加新 import | 中 ⚠️ | 评估是否引入新依赖 |

### 验证脚本模板

每个项目在 `dev/probes/` 下建验证脚本，改完代码后运行：

```python
# dev/probes/_probe_check_compatibility.py
from transform import 处理数据

模拟数据 = [("2026-01-01", "北京店", 100.0, 50)]
结果 = 处理数据(模拟数据, {})
assert isinstance(结果, list), "返回值必须是列表"
assert len(结果[0]) == 4, "输出列数必须为4"
print("✓ 兼容性验证通过")
```

## 六、调用链追溯

### 大框中的标注格式

```
║ ★ ★ ★ 调用链追溯 ★ ★ ★
║
║ ★ 导出函数调用位置：
║   处理数据() → main.py:跑()
║   处理数据() → post/report.py:生成报表()
║
║ ★ 被调后的处理逻辑：
║   处理数据()：
║     输入：[日期, 门店, 金额, 数量]
║     处理：过滤金额>0 → 按门店分组求和
║     输出：[日期, 门店, 汇总金额, 总数量]
║     副作用：无
║
║ ★ 外部函数引用：
║   连数据库.py → 查数据()
║   记日志.py → 日志()
║
║ ★ 如果改函数名，必须同步改：main.py, post/report.py
```

**四要素缺一不可：** 每个导出函数必须标注 **输入→处理→输出→副作用**。副作用包括：写了文件、改了数据库、发了请求、打了日志。

### 改接口的流程

```
1. 大框里查"被哪些文件调用了"
2. 改这个文件的函数名
3. 依次同步更新所有引用处
4. 运行验证脚本确认
```

## 七、迁移步骤（复制构建法）

不改当前运行版本，复制一份重组，验证后再切换：

```
1. cp -r 原项目/ 项目2/        ← 复制副本
2. 在项目2/ 里重组结构           ← 移动文件到子目录
3. 更新 main.py 里的路径引用
4. py main.py 试跑             ← 验证副本
5. mv 原项目/ archive/          ← 原版归档
6. mv 项目2/ 原项目名            ← 副本上位
```

**关键原则：不删原版，副本验证，最后改名完成切换。**

|------|-----------|--------|

## 九、Token 优化（AI 参与时压缩输入）

核心思路：利用模块化 + ★标记，AI 只需看一个文件。

| 要改什么 | 只发这个文件 |
|----------|------------|
| 改处理逻辑 | `★ transform.py` |
| 改配置参数 | `config.py` |
| 改取数方式 | `extract.py` |
| 改输出方式 | `load.py` |

**大框 = AI 的唯一上下文。** 大框包含调用关系+接口契约+调用链，AI 不需要读其他文件。★标记精确到行，AI 直接定位不改无关代码。

## 十、操作流程（生成/修改/验收的强制步骤）

> AI 生成、修改、验收代码时必须按步骤走，不能跳步。

### 10.1 生成代码流程

新建文件或新模块时，按顺序执行：

```
□ 第1步：读规范
   确认文件头部大框格式（第三节）、代码风格（第四节）

□ 第2步：写文件头部大框
   包含：文件名+说明、调用关系、数据流向、★/╳区域

□ 第3步：写全文逻辑说明
   一段中文说清：做了什么、上下游是谁、数据怎么流

□ 第4步：写变更记录
   第一行：当前日期 | AI | 初始版本

□ 第5步：写代码（逐条执行第四节）
   □ import 三组分隔 + 每行注释
   □ 每个函数中文 docstring
   □ 变量名中英混用（4.3 规则）
   □ ═══ 分步标记
   □ ★/╳ 修改点标记
   □ 错误处理（具体异常+中文信息）
   □ 魔法值提取为常量
   □ 早返回，不嵌套

□ 第6步：加接口契约（第五节）
   函数签名、入参出参格式、数据结构

□ 第7步：加调用链追溯（第六节）
   导出函数调用位置 + 输入→处理→输出→副作用 + 外部引用

□ 第8步：对照禁止事项检查（4.10）
```

### 10.2 修改代码流程

改现有文件时，按顺序执行：

```
□ 第1步：读文件头部大框
   确定 ★ 能改的区域 和 ╳ 不能动的区域

□ 第2步：读接口契约
   确认函数签名、入参出参格式绝对不能改

□ 第3步：读调用链追溯
   确认这个函数被哪些文件调用了、引用了哪些外部函数

□ 第4步：改代码
   只改 ★ 标注的区域，不动 ╳ 标注的区域

□ 第5步：更新调用链追溯
   改了函数名 → 同步更新所有引用处
   改了处理逻辑 → 更新"输入→处理→输出→副作用"描述

□ 第6步：运行验证脚本
   python dev/probes/_probe_check_compatibility.py
   确认接口没有被破坏

□ 第7步：更新版本号
   config.py 或文件头部：主版本.次版本（如 1.0 → 1.1）

□ 第8步：追加变更记录
   在变更记录区块追加一行，禁止覆盖历史

□ 第9步：对照禁止事项检查（4.10）
```

### 10.3 验收流程（4层检查）

代码审核时，按层次走：

```
□ 第1层：语法检查
   python 语法正确，无报错
   所有 import 的模块已安装

□ 第2层：规范检查（对照本规范全篇）
   □ 文件头部有大框？内容完整？
   □ 有全文逻辑说明？
   □ 有变更记录？格式正确？
   □ import 三组分隔 + 每行注释？
   □ 每个函数有中文 docstring？
   □ 变量名中英混用正确？
   □ 有 ★/╳ 标注？
   □ 有接口契约？
   □ 调用链有四要素（输入→处理→输出→副作用）？
   □ 没有魔法值？
   □ 错误处理有中文信息？
   □ 调试文件没放根目录？

□ 第3层：兼容性检查
   函数签名未被修改
   入参出参格式未被修改
   数据结构未被修改
   运行验证脚本通过

□ 第4层：逻辑检查
   修改后的逻辑符合预期
   边界情况已处理（空数据、异常数据）
   错误信息是中文
```

## 参考示例
