# project-structure-standard — Python 项目全链路规范

## 一句话

散乱项目按统一模板重组 + 每个文件写清接口契约和调用链，AI 只读一个文件就能安全改代码。

---

## 适用场景

- 项目根目录几十上百个文件混在一起，分不清生产代码和调试产物
- AI 改了一个文件，不知道会不会改坏其他模块
- 你要快速找到：入口在哪、配置在哪、核心逻辑在哪
- 多个项目需要统一的目录和代码规范

## 解决的问题

| 问题 | 方案 |
|------|------|
| 文件散落根目录 | 按职责划分子目录 |
| 改一个文件改坏其他模块 | 接口契约锁定签名+数据格式 |
| 不知道谁调了谁 | 调用链追溯标注 |
| 配置散落各处 | 集中到 `config.py` |
| 调试探针混入生产 | 隔离到 `dev/` |
| 不知道哪里能改 | `★` 标记标明修改点 |

---

## 一、通用目录模板

```
项目名/
│
├── core/                  ★ 不动：核心框架/基类
├── adapters/              ★ 不动：平台适配器
├── post/                  ★ 不动：后处理/回调
├── dashboard/             ★ 不动：看板/可视化
│
├── tools/                 ★ 独立功能工具
│
├── bin/                   ← 批处理/快捷脚本
├── config/                ← 配置文件（密钥、cookies）
│
├── dev/                   ← 调试探针/截图（不参与生产）
│   ├── probes/            ← _probe*.py
│   ├── debug/             ← debug*.py
│   ├── records/           ← 运行记录
│   ├── explores/          ← explore*.py test*.py
│   └── screenshots/       ← *.png
│
├── main.py                ★ 调度入口
├── config.py              ★ ★ ★ 你改配置的地方 ★ ★ ★
│
├── data/                  ← 数据文件（gitignore）
├── logs/                  ← 运行日志（gitignore）
│
└── README.md
```

### 核心规则

1. **根目录只放调度入口** — `main.py` 是唯一根级脚本
2. **配置只改 `config.py`** — 路径、参数、开关全部集中
3. **调试文件不进根目录** — 一律放入 `dev/`
4. **★ 标记告诉你哪里能改** — 带 `★` 的文件是你会碰到的
5. **后处理归入 `post/`**，独立功能归入 `tools/`

---

## 二、★ 标记系统

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `★ ★ ★` | 唯一配置修改点 | `config.py` 里的配置变量 |
| `★` | 看懂了可以改 | `main.py` 的调度函数 |
| `★ 不动` | 框架代码别碰 | `core/` `adapters/` 等 |
| `╳ 不要动` | 绝对不要改 | 接口签名、数据结构 |

## 三、文件头部规范

### 3.1 头部大框（每个 .py 文件必须有）

```
╔══════════════════════════════════════════╗
║ 文件名.py — 一句话说明                    ║
║                                           ║
║ 调用关系：被谁调用 → 本文件 → 调用了谁    ║
║ 数据流向：输入从哪来 → 处理 → 输出到哪去  ║
║                                           ║
║ ★ 接口契约：                              ║
║   函数名(入参) → 出参                     ║
║   入参格式 / 出参格式                     ║
║                                           ║
║ ★ 调用链：                                ║
║   导出函数 → 被谁调用                     ║
║   引用了哪些外部函数                      ║
║                                           ║
║ ★ 能改的区域：xxx                         ║
║ ╳ 不能动的区域：xxx                       ║
╚══════════════════════════════════════════╝
```

### 3.2 变更记录（大框之后、代码之前）

```python
# ═══════════════════════════════════════
# 变更记录
# ═══════════════════════════════════════
# 日期       | 作者 | 改了啥
# ───────────────────────────────────────
# 2026-06-05 | AI   | 初始版本
```

每次修改追加一行，禁止覆盖历史。

### 3.3 分步标记

```python
# ═══════════════════════════════════════════════════
# 第X步：这段代码在干嘛
# ═══════════════════════════════════════════════════
```

## 四、代码风格

### 4.1 import 写注释

注释说这个库在**当前文件**里具体干嘛用，不是库的标准文档。

```python
import os          # 操作用户文件夹（比如切目录）
import sys         # 控制 Python 搜索路径
```

### 4.2 函数写中文 docstring

```python
def 函数名(参数1, 参数2):
    """
    一句话说明干嘛。

    参数：
        参数1：含义，示例值
        参数2：含义，示例值

    返回：返回什么，什么格式

    举例：函数名(值1, 值2) → 返回什么
    """
```

### 4.3 变量命名：中英混用

| 层次 | 用中文 | 用英文 |
|------|--------|--------|
| 业务逻辑 | ✅ 函数名、变量名、参数名 | ❌ |
| Python 语法 | ❌ | ✅ 关键字、内置函数、特殊方法 |
| 第三方库调用 | ❌ | ✅ 方法名、属性名 |
| 数据/配置 | ❌ | ✅ SQL、键名、列名 |

```python
def 取数据(配置, 日期):
    sql = "SELECT * FROM 销售表"
    原始数据 = pd.read_sql(sql, 配置["连接"])  # 库方法保留英文
    return 原始数据.groupby("店铺").sum()
```

### 4.4 禁止事项

- ❌ 函数没有 docstring
- ❌ 配置硬编码在代码里
- ❌ 调试文件放在根目录
- ❌ try 块里没有中文错误信息
- ❌ 一段逻辑从头到尾没有注释

## 五、接口契约（零上下文修改的兼容性保障）

### 核心原则

每个文件的大框里标注接口契约，AI 只读一个文件就知道哪些能改、改了会不会破坏其他模块。

### 三层保障

| 层 | 保障什么 | 怎么写 |
|----|----------|--------|
| 第1层 | 函数签名不变 | 大框标注 `def 函数名(参数)` → 只能改函数体 |
| 第2层 | 数据格式不变 | 大框标注列结构：`行[0]=日期, 行[1]=门店` |
| 第3层 | 行为兼容验证 | `dev/probes/_probe_check_compatibility.py` |

### 改动风险等级

| 改动类型 | 风险 | 预防 |
|----------|------|------|
| 改函数体内部逻辑 | 低 ✅ | 接口契约锁定签名 |
| 改函数名/入参/返回格式 | 高 ❌ | 大框标注"不能改" |
| 改配置键名 | 高 ❌ | config.py 头部写死"不能改" |
| 加新 import | 中 ⚠️ | 评估是否引入新依赖 |

### 验证脚本模板

每个项目在 `dev/probes/` 下建验证脚本，改完代码后运行：

```python
# dev/probes/_probe_check_compatibility.py
from transform import 处理数据

模拟数据 = [("2026-01-01", "北京店", 100.0, 50)]
结果 = 处理数据(模拟数据, {})
assert isinstance(结果, list), "返回值必须是列表"
assert len(结果[0]) == 4, "输出列数必须为4"
print("✓ 兼容性验证通过")
```

## 六、调用链追溯

### 大框中的标注格式

```
║ ★ 导出函数调用位置：
║   处理数据() → main.py:跑()
║   处理数据() → post/report.py:生成报表()
║
║ ★ 外部函数引用：
║   连数据库.py → 查数据()
║   记日志.py → 日志()
║
║ ★ 如果改函数名，必须同步改：main.py, post/report.py
```

### 改接口的流程

```
1. 大框里查"被哪些文件调用了"
2. 改这个文件的函数名
3. 依次同步更新所有引用处
4. 运行验证脚本确认
```

## 七、迁移步骤（复制构建法）

不改当前运行版本，复制一份重组，验证后再切换：

```
1. cp -r 原项目/ 项目2/        ← 复制副本
2. 在项目2/ 里重组结构           ← 移动文件到子目录
3. 更新 main.py 里的路径引用
4. py main.py 试跑             ← 验证副本
5. mv 原项目/ archive/          ← 原版归档
6. mv 项目2/ 原项目名            ← 副本上位
```

**关键原则：不删原版，副本验证，最后改名完成切换。**

## 八、增删改操作表

| 操作 | 改哪个文件 | 怎么改 |
|------|-----------|--------|
| 加一个新项目 | `main.py` 的项目列表 | 加一行 |
| 改项目配置 | `config.py` | 改变量值 |
| 改业务逻辑 | `main.py` → `跑()` | 改函数内容 |
| 换数据源 | `config.py` + `extract.py` | 改配置+改取数 |
| 换输出方式 | `load.py` | 改写入逻辑 |
| 加调试探针 | `dev/probes/` 下新建 | 用探针模板 |

## 九、Token 优化（AI 参与时压缩输入）

核心思路：利用模块化 + ★标记，AI 只需看一个文件。

| 要改什么 | 只发这个文件 |
|----------|------------|
| 改处理逻辑 | `★ transform.py` |
| 改配置参数 | `config.py` |
| 改取数方式 | `extract.py` |
| 改输出方式 | `load.py` |

**大框 = AI 的唯一上下文。** 大框包含调用关系+接口契约+调用链，AI 不需要读其他文件。★标记精确到行，AI 直接定位不改无关代码。
